# М4 — Налоговый календарь

## Статус

Backend pipeline готов

Модуль скачивает официальный налоговый календарь ФНС, парсит налоговые события, собирает дневной календарный датасет М4 и feature dataset после анализа `ml/notebooks/module_4.ipynb`

Feature builder содержит только устойчивую расчетную часть из ноутбука. Графики, демонстрационная STL-декомпозиция М1 и презентационные проверки в backend не перенесены

## Источник

Основной источник:

```text
https://www.nalog.gov.ru/opendata/7707329152-kalendar/
```

Это официальный набор открытых данных ФНС `Налоговый календарь`

Используются:

```text
XML-релизы календаря
XSD-структура набора данных
паспорт набора открытых данных
```

На момент реализации актуальный XML-релиз:

```text
https://data.nalog.ru/opendata/7707329152-kalendar/data-20260122-structure-20140228.xml
```

## Запуск

```bash
python3 backend/src/pipelines/m4_pipeline.py
```

XML и HTML разбираются стандартной библиотекой Python

Для сохранения `m4_features.parquet` используется `pyarrow`, потому что стандартная библиотека Python не умеет сохранять Parquet. Поэтому запускать pipeline лучше из `.venv`, как М1-М3:

```bash
.venv/bin/python backend/src/pipelines/m4_pipeline.py
```

`event_type` — техническая классификация события

`source_url` — ссылка на XML-релиз ФНС

## Колонки m4_dataset.csv

`date` — календарная дата

`is_tax_payment_day` — есть ли на дату налоговое платежное событие

`is_tax_reporting_day` — есть ли на дату событие сдачи отчетности

`is_notification_day` — есть ли на дату событие подачи уведомления

`tax_events_count` — количество всех налоговых событий на дату

`tax_payment_events_count` — количество платежных событий

`tax_reporting_events_count` — количество событий отчетности

`notification_events_count` — количество событий уведомлений

`other_events_count` — количество событий, которые не попали в базовую классификацию

`days_to_next_tax_payment` — дней до ближайшего налогового платежного события

`days_since_prev_tax_payment` — дней после предыдущего налогового платежного события

`is_month_end` — дата является последним днем месяца

`is_quarter_end` — дата является последним днем квартала

`is_year_end` — дата является последним днем года

## Колонки m4_features.csv

`m4_features.csv` и `m4_features.parquet` строятся из `m4_dataset.csv`

`Tax_Pre_Flag` — в ближайшие 1-3 дня будет налоговый платеж, но сегодня платежа нет

`Tax_Active_Flag` — сегодня есть налоговый платеж

`Tax_Post_Flag` — в предыдущие 1-3 дня был налоговый платеж, но сегодня платежа нет

`Tax_Week_Flag` — дата попадает в налоговое окно `pre / active / post`

`Tax_Day_Strict` — дата платежа или соседний день рядом с платежом

`Regime_Post_ENP` — дата после реформы ЕНП, начиная с 01.01.2023

`tax_pressure` — взвешенная интенсивность налоговой нагрузки

`tax_pressure_smoothed` — центрированное скользящее среднее `tax_pressure` за 7 дней

`tax_proximity` — экспоненциальная близость к ближайшей налоговой дате

`MAD_tax_pressure` — MAD-score для `tax_pressure` за daily-окно 3 года

`MAD_tax_proximity` — MAD-score для `tax_proximity` за daily-окно 3 года

`Seasonal_Factor_raw` — несглаженный сезонный мультипликатор

`Seasonal_Factor` — сглаженный сезонный мультипликатор за 5 дней

## Принятые решения

Downloader сохраняет паспорт набора открытых данных, XSD и все найденные XML-релизы

Это сделано потому, что актуальный XML обычно покрывает только один календарный год. Для исторического анализа нужны предыдущие релизы ФНС

Parser делает одну строку на одно текстовое событие внутри налогового дня. Название налога берется из ближайшего заголовка внутри HTML-фрагмента дня

`event_type` определяется по тексту:

```text
payment_deadline — есть слова про уплату, перечисление или внесение платежа
reporting_deadline — есть слова про представление декларации, расчета, сведений или отчетности
notification_deadline — есть слова про уведомление
mixed — в одном событии есть платежный и отчетный или уведомительный смысл
other — событие не попало в базовые правила
```

`m4_dataset.csv` строится как дневной календарь от первой до последней даты, найденной в релизах ФНС

На backend-этапе не добавляются ML-признаки. Датасет нужен аналитику, чтобы проверить полезность налогового календаря и согласовать feature builder

После анализа `ml/notebooks/module_4.ipynb` добавлен `m4_feature_builder.py`

Флаги налогового окна считаются по окну 3 дня до и после налогового платежа. Это эвристика аналитика, которую нужно явно учитывать в финальной методологии

`tax_pressure` считается как сумма базового платежного дня и бонусов за концентрацию событий, конец квартала и конец года. Это не сумма налоговых платежей в рублях, потому что ФНС публикует календарь сроков, а не объемы платежей

`Seasonal_Factor` является контекстным мультипликатором для будущего агрегационного слоя, но сам по себе не является LSI

`m4_features.parquet` сохраняет те же признаки в типизированном формате для аналитики и ML

## Ограничения

Календарь ФНС содержит даты и текст событий, но не содержит суммы налоговых платежей. Поэтому М4 на текущем этапе показывает календарное давление на ликвидность, а не объем изъятия ликвидности

Классификация `event_type` сделана простыми правилами по тексту. Если аналитик подтвердит более точные правила, их нужно вынести в `m4_feature_builder.py`

Актуальный XML ФНС за 2026 год содержит некорректное объявление кодировки: в декларации указан `windows-1251`, но файл фактически отдается как UTF-8. Parser учитывает это через fallback чтения

`MAD_tax_pressure` и `MAD_tax_proximity` пустые в начале истории, пока не накоплено минимальное daily-окно. Это ожидаемо

`Tax_Week_Flag` покрывает значительную долю дней, потому что в календаре много разных налоговых дат. Для модели важнее смотреть его вместе с непрерывными признаками `tax_pressure` и `tax_proximity`

## Что не делается в М4

Не считаются:

```text
налоговые stress flags
LSI
финальные стресс-сигналы
```

Эти части добавляются после анализа данных и согласования признаков
