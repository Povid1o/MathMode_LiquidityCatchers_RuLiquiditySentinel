# М3 — Размещение ОФЗ

## Статус

Backend pipeline готов

Модуль собирает официальные результаты аукционов Минфина по размещению государственных ценных бумаг, формирует базовый датасет М3 и отдельный feature dataset после анализа `ml/notebooks/module_3.ipynb`

## Источник

Основной источник:

```text
https://minfin.gov.ru/ru/document/
```

Используется поиск документов Минфина по запросу:

```text
Результаты проведенных аукционов по размещению государственных ценных бумаг
```

На момент реализации источник дает XLSX-файлы за 2016-2026 годы

ЦБ из ТЗ рассматривается как дополнительный источник подтверждения, но не используется в базовом pipeline М3, потому что результаты аукционов в структурированном виде публикует Минфин

## Колонки

`auction_date` — дата аукциона

`published_date` — дата публикации документа Минфина

`document_title` — название документа Минфина

`auction_format` — формат размещения, если есть в файле Минфина, например `Аукцион` или `ДРПА`

`issue` — код выпуска ОФЗ

`security_type` — тип бумаги, например `ОФЗ-ПД`, `ОФЗ-ПК`, `ОФЗ-ИН`

`maturity_date` — дата погашения выпуска

`days_to_maturity` — количество дней до погашения на дату аукциона

`offered_amount` — объем предложения, млн руб

`demand_amount` — совокупный объем спроса по номиналу, млн руб

`placed_amount` — объем размещения по номиналу, млн руб

`proceeds_amount` — объем выручки, млн руб

`cutoff_price` — цена отсечения, % от номинала

`weighted_average_price` — средневзвешенная цена, % от номинала

`cutoff_yield` — доходность по цене отсечения, % годовых

`weighted_average_yield` — доходность по средневзвешенной цене, % годовых

`official_coefficient` — коэффициент из файла Минфина

`cover_ratio` — технический показатель спроса к предложению

`placement_ratio` — технический показатель размещения к предложению

`source_url` — ссылка на страницу документа Минфина

`source_file` — имя XLSX-файла Минфина

## Принятые решения

`cover_ratio` считается как:

```text
demand_amount / offered_amount
```

Это соответствует формулировке ТЗ для М3: cover ratio = спрос / предложение

В файлах Минфина после 2025 года официальный коэффициент называется `Коэффициент удовлетворения спроса на аукционе` и считается иначе:

```text
placed_amount / demand_amount
```

Поэтому в датасете сохранены разные поля:

`official_coefficient` — значение как в официальном файле Минфина

`cover_ratio` — расчет по ТЗ, спрос к предложению

`placement_ratio` — размещение к предложению

Это позволяет аналитику сравнить методики и выбрать нужную для feature builder

## Feature dataset

`m3_features.csv` и `m3_features.parquet` строятся из `m3_dataset.csv`

Аукционы агрегируются по дню:

```text
demand_amount — сумма спроса за день
offered_amount — сумма предложения за день
placed_amount — сумма размещения за день
weighted_yield — средневзвешенная доходность за день по placed_amount
```

Дальше считаются признаки:

```text
cover_ratio
yield_spread
Flag_Nedospros
Flag_Perespros
MAD_score_cover
MAD_score_yield_spread
```

Дневной `cover_ratio` считается как:

```text
demand_amount / offered_amount
```

Если предложение равно нулю, используется запасной расчет:

```text
demand_amount / placed_amount
```

`yield_spread` на этом этапе считается как разница дневной средневзвешенной доходности с предыдущим днем аукционов:

```text
weighted_yield - previous_weighted_yield
```

Это приближение из аналитического ноутбука. В production-версии вместо него лучше использовать спред к точке кривой бескупонной доходности или другой согласованный бенчмарк по сроку бумаги

Флаги считаются так:

```text
Flag_Nedospros = 1, если cover_ratio < 1.2
Flag_Perespros = 1, если cover_ratio > 2.0
```

`MAD_score_cover` и `MAD_score_yield_spread` считаются через скользящую медиану и MAD за 3 календарных года. Минимальное значение MAD ограничено снизу `0.05`, чтобы не получать математические выбросы на плоских участках данных

`m3_features.parquet` сохраняет те же признаки в типизированном формате для аналитики и ML. Для записи используется `pyarrow`, потому что стандартная библиотека Python не умеет сохранять Parquet

## Ограничения

История сейчас начинается с 2016 года, потому что найденная страница Минфина содержит документы 2016-2026

Некоторые поля пустые в части строк. Это ожидаемо:

```text
demand_amount может отсутствовать для ДРПА
cutoff_yield и weighted_average_yield могут отсутствовать для отдельных ОФЗ-ПК или несостоявшихся аукционов
auction_format отсутствует в старых файлах, где такой колонки еще не было
```

`ofz_auctions.csv` и `m3_dataset.csv` на текущем backend-этапе почти совпадают по смыслу. Разделение оставлено для общей архитектуры проекта: parser отвечает за source CSV, dataset builder отвечает за модульный датасет

## Что не делается в М3

Не считаются:

```text
yield_spread_to_curve
m3_signal
LSI
SHAP
финальные стресс-сигналы
```

Эти части относятся к последующим задачам аналитика, ML или агрегационного слоя
