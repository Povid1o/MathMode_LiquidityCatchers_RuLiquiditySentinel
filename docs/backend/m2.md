# М2 — Аукционы репо ЦБ

## Статус

Базовый backend pipeline готов

Модуль собирает итоги аукционов репо ЦБ и ключевую ставку, формирует базовый датасет М2 для дальнейшей аналитики и ML

Feature builder для М2 добавлен по результатам анализа в `ml/notebooks/model_2.ipynb`

Модуль теперь формирует:

```text
m2_dataset.csv
m2_features.csv
```

## Почему есть сводные и дневные raw-файлы

Страница ЦБ по репо работает в двух режимах

Если запрашивать диапазон дат, ЦБ отдает сводную таблицу:

```text
date
auction_type
term_days
auction_time
total_deals_volume
weighted_average_rate
settlement_code
```

Если запрашивать один день, ЦБ отдает детальную карточку аукциона:

```text
demand_volume
cutoff_rate
min_rate
max_rate
limit_deals_volume
weighted_average_limit_rate
first_leg_date
second_leg_date
```

Поэтому pipeline сначала скачивает сводную таблицу, получает даты реальных аукционов, а затем скачивает дневные страницы только для этих дат

Дневные HTML сохраняются в `data/raw/repo/daily/` и игнорируются git, потому что это большой воспроизводимый кеш

## Колонки repo.csv

`repo.csv` содержит:

```text
date
auction_type
term_days
auction_time
total_deals_volume
weighted_average_rate
settlement_code
demand_volume
cutoff_rate
min_rate
max_rate
limit_deals_volume
weighted_average_limit_rate
first_leg_date
second_leg_date
cover_ratio
```

`cover_ratio` считается как:

```text
demand_volume / total_deals_volume
```

В источнике ЦБ по репо нет отдельной колонки предложения. Поэтому на backend-этапе используется спрос к объему заключенных сделок. Если аналитик выберет другую методику, это нужно будет скорректировать в `m2_feature_builder.py`

## Колонки m2_dataset.csv

`m2_dataset.csv` содержит все поля `repo.csv` и поле:

```text
key_rate
```

Ключевая ставка добавляется из отдельного источника ЦБ

## Колонки m2_features.csv

`m2_features.csv` и `m2_features.parquet` строятся из `m2_dataset.csv` и содержат только 7-дневные аукционы с 2010 года

Базовые поля сохраняются из `m2_dataset.csv`, затем добавляются признаки:

```text
rate_for_spread
rate_spread
Flag_Demand
MAD_score_cover
MAD_score_rate_spread
```

`rate_for_spread` выбирается так:

```text
cutoff_rate, если он есть
weighted_average_rate, если cutoff_rate отсутствует
```

`rate_spread` считается как:

```text
rate_for_spread - key_rate
```

`Flag_Demand` равен 1, если:

```text
cover_ratio > 2.0
```

`MAD_score_cover` и `MAD_score_rate_spread` считаются через скользящую медиану и MAD за 3 календарных года. Минимальное значение MAD ограничено снизу `0.05`, чтобы не получать математические выбросы на плоских участках данных

`cover_ratio` в feature dataset пересчитывается по методике аналитика:

```text
demand_volume / total_deals_volume
```

Если объем сделок меньше `1000` млн руб. или данных недостаточно, значение считается нейтральным:

```text
cover_ratio = 1.0
```

Сверху `cover_ratio` ограничивается значением `10.0`

## Принятые решения

`repo.csv` остается source-level датасетом по аукционам репо

`m2_dataset.csv` объединяет репо и ключевую ставку

`m2_features.csv` является feature dataset для аналитика и ML. В нем уже есть фильтрация 7-дневных аукционов, расчет спреда, флаг переспроса и MAD-нормализация

`m2_features.parquet` сохраняет тот же набор признаков в типизированном формате для удобной загрузки в аналитике и ML. Для записи используется `pyarrow`, потому что стандартная библиотека Python не умеет сохранять Parquet

Дневной raw-кеш не пушится, потому что он занимает много файлов и полностью воспроизводится downloader'ом

`cover_ratio` добавлен как базовое техническое поле из ТЗ, а не как ML-фича

## Ограничения

Не все исторические строки имеют дневные детальные поля. Это связано с особенностями официальных страниц ЦБ:

```text
старые аукционы могут иметь неполные данные
несостоявшиеся аукционы могут не иметь спроса или ставок
детальная карточка не всегда содержит все поля
```

Ключевая ставка доступна только с периода, покрытого источником ЦБ

## Что не делается в М2

Пока не считаются:

```text
лаги
m2_signal
LSI
SHAP
финальные стресс-сигналы
```

Эти части относятся к последующим задачам аналитика, ML или агрегационного слоя
